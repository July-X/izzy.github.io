`MiMo TTS Studio` 支持三种音色来源：预置音色（系统内置）、设计音色（文本描述生成）、克隆音色（参考音频复制）。三种模式的数据结构、API 调用方式完全不同，但需要在同一个角色管理界面中统一操作。

## 数据模型：Character 绑定一切

核心设计是 `Character`（角色）作为音色的载体，一个角色 = 一个音色配置 + 一组台词：

```typescript
export type VoiceSource = 'preset' | 'voicedesign' | 'voiceclone'

export interface Character {
  id: string
  name: string
  voiceId: string                    // 预置模式的音色 ID
  voiceSource: VoiceSource           // 音色来源模式
  voiceDesignPrompt: string          // 设计模式的文本描述
  cloneVoicePath: string             // 克隆模式的参考音频路径
  cloneVoiceMime: string             // 克隆音频 MIME 类型
  styleTags: string[]                // 风格标签（方言、情感）
  speed: number                      // 语速 0.5-2.0
  lines: Line[]                      // 关联台词
}
```

## 三种模式的 API 调用差异

### 预置音色

直接传 `voiceId`，API 从内置音色库匹配：

```typescript
// 模式 A：预置音色
{
  model: 'MiMo-V2.5-TTS',
  voice_id: character.voiceId,        // 如 'female-tianmei'
  text: line.text,
  speed: character.speed
}
```

### 设计音色

不传 `voiceId`，改传 `voice` 对象的 `customization` 字段，API 根据文本描述动态生成音色：

```typescript
// 模式 B：设计音色
{
  model: 'MiMo-V2.5-TTS',
  voice: {
    customization: character.voiceDesignPrompt  // 如 "温柔的年轻女性，语速偏慢"
  },
  text: line.text,
  speed: character.speed
}
```

### 克隆音色

传 `voice` 对象的 `clone` 字段，包含参考音频的 Base64 编码和 MIME 类型：

```typescript
// 模式 C：克隆音色
{
  model: 'MiMo-V2.5-TTS',
  voice: {
    clone: {
      audio: base64Audio,              // 参考音频 Base64
      mime: character.cloneVoiceMime   // 'audio/wav'
    }
  },
  text: line.text,
  speed: character.speed
}
```

## 统一调用入口

Service 层用一个方法统一处理三种模式，根据 `voiceSource` 分支：

```javascript
// electron/service/tts.js
async synthesize(character, line) {
  const body = {
    model: 'MiMo-V2.5-TTS',
    text: line.text,
    speed: character.speed
  }

  switch (character.voiceSource) {
    case 'preset':
      body.voice_id = character.voiceId
      break
    case 'voicedesign':
      body.voice = { customization: character.voiceDesignPrompt }
      break
    case 'voiceclone':
      const audio = await fs.readFile(character.cloneVoicePath)
      body.voice = {
        clone: {
          audio: audio.toString('base64'),
          mime: character.cloneVoiceMime
        }
      }
      break
  }

  return this.callAPI(body)
}
```

克隆模式需要额外读取本地音频文件并编码为 Base64，这是 Electron 的 `fs` 能力发挥作用的地方——Web 应用无法直接读取本地文件。

## 持久化：voice.json

音色配置独立于台词数据，持久化到 `voice.json`：

```typescript
export interface VoiceConfig {
  voiceId: string
  voiceSource: VoiceSource
  voiceDesignPrompt: string
  cloneVoicePath: string           // 存绝对路径，跨机器需重新指定
  cloneVoiceMime: string
  styleTags: string[]
  speed: number
}
```

保存时序列化为 JSON，加载时反序列化并校验。克隆模式的 `cloneVoicePath` 存绝对路径，意味着项目文件夹复制到另一台机器后，克隆音色需要重新指定音频文件。

## 风格标签系统

`styleTags` 是一个字符串数组，支持方言和情感标签的自由组合：

```typescript
// 方言标签：'Sichuan' | 'Cantonese' | 'Dongbei' | ...
// 情感标签：'happy' | 'sad' | 'angry' | 'gentle' | ...
character.styleTags = ['Sichuan', 'happy']
```

标签传递给 API 的 `style_tags` 字段，影响语音的方言口音和情感表达。一个角色可以同时有方言 + 情感的复合标签。

## 设计取舍

**为什么音色配置和台词分开存？**
- 同一个音色可能配多组台词（不同场景的配音脚本）
- 台词变化频繁，音色配置相对稳定
- 分开存减少序列化/反序列化的数据量

**为什么克隆模式存绝对路径？**
- 音频文件可能很大（几 MB ~ 几十 MB），不适合嵌入 JSON
- 用户可能随时修改参考音频，存路径可以实时读取最新版本
- 代价是跨机器迁移需要重新指定

> 统一接口、分支实现。三种模式的差异被封装在 Service 层，前端和 Controller 无需关心音色来源。
