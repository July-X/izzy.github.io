SLG 游戏有明显的流量波峰——每天的定时活动、跨服战开启、新服开服。如果不做背压控制，瞬间涌入的请求会打垮服务。`slg-go` 实现了多级背压控制，从连接层到业务层逐级保护。

## 三级背压

### 第一级：Gate 连接限流

Gate 层限制最大连接数，新连接直接拒绝：

```go
// gate/server.go
func (s *Server) handleConnection(conn net.Conn) {
    if s.connCount.Load() >= s.maxConns {
        conn.Write([]byte("server_full"))
        conn.Close()
        return
    }
    s.connCount.Add(1)
    defer s.connCount.Add(-1)
    s.serve(conn)
}
```

### 第二级：Logic 队列背压

Logic 层用 channel 做请求队列，队列满时返回 busy：

```go
// logic/shard.go
type Shard struct {
    requestCh chan *Request
    queueSize int32
}

func (s *Shard) Handle(req *Request) error {
    if len(s.requestCh) >= cap(s.requestCh)*90/100 {
        return ErrServerBusy  // 队列 90% 满，拒绝
    }
    s.requestCh <- req
    return nil
}
```

客户端收到 `server_busy` 后指数退避重试：

```gdscript
# Godot 客户端
func _retry_with_backoff(attempt: int) -> void:
    var delay = min(0.1 * pow(2, attempt), 5.0)  # 最大 5 秒
    await get_tree().create_timer(delay).timeout
    send_request()
```

### 第三级：Battle CPU 限流

Battle 层监控 CPU 使用率，超过阈值时延迟处理：

```go
// battle/limiter.go
func (s *Service) shouldThrottle() bool {
    usage := s.cpuMonitor.Usage()
    return usage > 0.85  // CPU 超过 85% 开始限流
}

func (s *Service) Handle(ctx context.Context, req *BattleRequest) (*BattleResult, error) {
    if s.shouldThrottle() {
        time.Sleep(time.Duration(rand.Intn(100)) * time.Millisecond)
    }
    return s.simulate(ctx, req)
}
```

随机延迟让请求分散到不同时间窗口，避免所有请求同时涌入。

## Prometheus 监控

每层暴露 Prometheus 指标：

```go
var (
    activeConns = prometheus.NewGauge(prometheus.GaugeOpts{
        Name: "gate_active_connections",
        Help: "当前活跃连接数",
    })
    requestQueueDepth = prometheus.NewGaugeVec(prometheus.GaugeOpts{
        Name: "logic_queue_depth",
        Help: "Logic 请求队列深度",
    }, []string{"shard"})
    battleLatency = prometheus.NewHistogram(prometheus.HistogramOpts{
        Name:    "battle_latency_seconds",
        Help:    "战斗计算延迟",
        Buckets: prometheus.ExponentialBuckets(0.01, 2, 10),
    })
)
```

## 自动扩容

基于 Prometheus 指标，通过 Kubernetes HPA 自动扩容：

```yaml
# k8s/hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: battle-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: battle
  minReplicas: 5
  maxReplicas: 30
  metrics:
    - type: Pods
      pods:
        metric:
          name: battle_latency_seconds_p99
        target:
          type: AverageValue
          averageValue: "500m"  # P99 延迟 > 0.5s 时扩容
```

Battle 层无状态，可以随意扩缩。Logic 层有状态，扩容需要数据迁移，通常固定分片数，不自动扩容。

## 开服应对

新服开服是最极端的场景。预热策略：

1. **提前扩容**：开服前 30 分钟扩容 Battle 到 30 实例
2. **排队系统**：Gate 层接入排队系统，按顺序放行
3. **分批放行**：每 10 秒放行 1000 人，避免瞬间打满

```go
// gate/queue.go
type Queue struct {
    waiting   []int64
    rate      int           // 每秒放行人数
    interval  time.Duration // 放行间隔
}

func (q *Queue) process() {
    ticker := time.NewTicker(q.interval)
    for range ticker.C {
        n := min(q.rate, len(q.waiting))
        for i := 0; i < n; i++ {
            playerID := q.waiting[0]
            q.waiting = q.waiting[1:]
            q.allow(playerID)
        }
    }
}
```

> 背压控制不是"限流"，是"优雅降级"。宁可让部分玩家等几秒，也不能让整个服务崩溃。
