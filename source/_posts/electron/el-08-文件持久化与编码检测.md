> MiMo TTS Studio 的数据存储设计非常独特——**不使用数据库，而是直接用文件系统来管理项目数据**。每个角色是一个文件夹，台词是纯文本文件。这种"角色即文件夹"的设计让项目天然可版本控制、跨设备迁移。本文拆解这个方案的实现细节，以及中文环境下最头疼的 **编码自动检测** 问题。

---

## Q1："角色即文件夹"的完整目录结构是什么样的？

**A：**

```
~/Documents/MiMo-Projects/我的游戏配音/          ← 项目根目录（用户自选）
├── NPC张三/                                      ← 角色文件夹 = 一个角色
│   ├── voice.json                                 ← 音色配置 (JSON)
│   ├── script.txt                                 ← 台词文本 (UTF-8, 一行一句)
│   ├── lines_meta.json                            ← 台词元数据 (JSON)
│   └── audio/                                     ← 生成的音频文件
│       ├── NPC张三_L01.wav                        ← 第 1 条台词的音频
│       ├── NPC张三_L02.wav                        ← 第 2 条
│       └── NPC张三_L05.wav                        ← 第 5 条（3、4 还没生成）
│
├── NPC李四/
│   ├── voice.json
│   ├── script.txt
│   ├── lines_meta.json
│   └── audio/
│       ├── NPC李四_L01.wav
│       └── ...
│
└── temp/                                          ← 临时文件（试听音频等）
```

### 每个文件的职责

| 文件 | 格式 | 内容 | 谁读写 |
|------|------|------|--------|
| `voice.json` | JSON | 音色 ID、模式、速度、风格标签、克隆路径 | characterStore |
| `script.txt` | 纯文本 UTF-8 | 台词原文，一行一句 | characterStore |
| `lines_meta.json` | JSON | 每条台词的 ID / 音频路径 / 状态 / 时长 | characterStore |
| `audio/*.wav` | 二进制 WAV | TTS 引擎生成的音频 | TTSService（主进程） |

### 为什么这样设计？

```
对比传统方案（SQLite / IndexedDB）：

✅ 文件系统方案：
  ├── Finder 里可以直接看到所有内容
  ├── git diff 能精确看到哪句台词改了
  ├── cp 整个文件夹就能备份/迁移一个角色
  ├── script.txt 可以用任何编辑器编辑
  └── 不需要数据库驱动、迁移脚本

❌ 数据库方案：
  ├── 需要专门的工具查看数据
  ├── git diff 显示的是整个二进制 blob 的变化
  ├── 备份需要导出功能
  ├── 离开应用无法编辑数据
  └── 需要处理 schema migration
```

对于 TTS 工具这种**以文件为核心**的应用来说，文件系统就是最好的数据库。

## Q2：启动时怎么从磁盘恢复角色列表？

**A：** 应用启动时扫描项目目录，重建内存状态：

```javascript
// characterStore.js — 启动扫描恢复

async function loadProject(projectPath) {
  projectPath.value = projectPath;

  // 1. 扫描子目录（每个子目录 = 一个角色）
  const entries = await readdir(projectPath);
  const dirs = entries.filter(e => e.isDirectory);

  const restoredChars = [];

  for (const dir of dirs) {
    if (dir.name.startsWith('.') || dir.name === 'temp' || dir.name === 'audio') continue;

    // 2. 读取 voice.json → 角色基础信息
    const voicePath = `${projectPath}/${dir.name}/voice.json`;
    let voiceConfig = {};
    try {
      const raw = await readFile({ filePath: voicePath });
      voiceConfig = JSON.parse(raw);
    } catch { /* 新目录可能还没有 voice.json */ }

    // 3. 读取 script.txt → 台词列表
    const scriptPath = `${projectPath}/${dir.name}/script.txt`;
    let lineTexts = [];
    try {
      const raw = await readTextFile({ filePath: scriptPath });  // 带编码检测！
      lineTexts = raw ? raw.split('\n').filter(l => l.trim()) : [];
    } catch { /* 同上 */ }

    // 4. 读取 lines_meta.json → 合并已生成的状态
    const metaPath = `${projectPath}/${dir.name}/lines_meta.json`;
    let metaData = [];
    try {
      const raw = await readFile({ filePath: metaPath });
      metaData = JSON.parse(raw);
    } catch { /* 首次使用可能还没有 meta */ }

    // 5. 组装为内存对象
    const ch = {
      id: generateId('char'),
      name: dir.name,
      voiceId: voiceConfig.voiceId || 'mimo_default',
      voiceSource: voiceConfig.voiceSource || 'preset',
      voiceDesignPrompt: voiceConfig.voiceDesignPrompt || '',
      cloneVoicePath: voiceConfig.cloneVoicePath || '',
      styleTags: voiceConfig.styleTags || [],
      speed: voiceConfig.speed || 1.0,
      lines: lineTexts.map((text, idx) => {
        const meta = metaData[idx] || {};
        return {
          id: meta.id || generateId('line'),
          text,
          audioPath: meta.audioPath || '',
          status: meta.status || 'pending',     // 恢复之前的进度！
          duration: meta.duration || 0,
        };
      }),
    };

    restoredChars.push(ch);
  }

  characters.value = restoredChars;
  if (restoredChars.length > 0) {
    currentCharacterId.value = restoredChars[0].id;
  }
}
```

**关键点：`lines_meta.json` 保存了生成状态和进度**——用户关闭应用再打开后，已经合成好的台词不会丢失，显示为 `done` 状态，未完成的仍然是 `pending`。

## Q3：防抖自动落盘是怎么实现的？

**A：** 每次用户操作（改音色、加台词、修改文本）都会触发状态变更，但不能每次都写磁盘——IO 太频繁了：

```javascript
// characterStore.js — 防抖落盘

let _saveTimer = null;
const SAVE_DEBOUNCE_MS = 1000;  // 1 秒无操作后才真正写入

function scheduleSave(ch) {
  if (_saveTimer) clearTimeout(_saveTimer);
  _saveTimer = setTimeout(async () => {
    _saveTimer = null;
    await saveCharacterToDisk(ch);
  }, SAVE_DEBOUNCE_MS);
}

// 在各个 mutation 函数中调用
async function updateLine(lineId, updates) {
  const ch = currentCharacter.value;
  const line = ch.lines.find(l => l.id === lineId);
  if (line) { Object.assign(line, updates); }
  scheduleSave(ch);  // ← 不是立刻保存，而是"安排"一次
}
```

**效果：**

```
用户快速输入 10 条台词（每条间隔 200ms）
  ↓ 触发 10 次 scheduleSave()
  ↓ 但前 9 次都被 clearTimeout 取消了
  ↓ 只有最后一次（第 10 次之后 1 秒没新操作）
  ↓ 真正执行 saveCharacterToDisk() —— 一次性写入全部变更

实际 IO 操作: 1 次
如果不用防抖: 10 次
```

### 应用退出时的兜底

```javascript
// 应用退出前强制保存
app.on('before-quit', async () => {
  if (_saveTimer) {
    clearTimeout(_saveTimer);        // 清除定时器
    _saveTimer = null;
  }
  // 立即保存所有有变更的角色
  for (const ch of characters.value) {
    await saveCharacterToDisk(ch);
  }
});
```

## Q4：编码检测是怎么回事？为什么中文环境必须处理？

**A：** 这是中文开发桌面应用的**经典坑**。用户导入的台词脚本可能来自各种来源：

| 来源 | 典型编码 |
|------|---------|
| Windows 记事本（默认保存） | GBK / GB2312 |
| Windows 记事本 "另存为 UTF-8" | UTF-8 BOM |
| Mac / Linux 文本编辑器 | UTF-8 无 BOM |
| 从网页复制的文字 | UTF-8 |
| Excel 导出的 CSV | 可能是 UTF-16 LE（Windows 区域） |

如果用错误的编码去读文件，中文会变成乱码：

```
原始文本: 你好，世界
GBK 文件被当作 UTF-8 读: 浣锛涓�涓���（乱码）
UTF-8 文件被当作 GBK 读: 姹濆锕屾仝湰（也是乱码）
```

### readTextFile 的完整实现

```javascript
// electron/controller/file.js — 主进程侧

async readTextFile(args = {}) {
  try {
    const buffer = await fsPromises.readFile(args.filePath);
    if (!buffer || buffer.length === 0) return '';

    // ===== 第一步：BOM 检测 =====
    // UTF-8 BOM: EF BB BF
    if (buffer[0] === 0xEF && buffer[1] === 0xBB && buffer[2] === 0xBF) {
      return buffer.toString('utf-8', 3);         // 跳过 BOM 前 3 字节
    }
    // UTF-16 LE BOM: FF FE
    if (buffer[0] === 0xFF && buffer[1] === 0xFE) {
      return buffer.toString('utf-16le', 2);        // 小端序
    }
    // UTF-16 BE BOM: FE FF
    if (buffer[0] === 0xFE && buffer[1] === 0xFF) {
      return buffer.toString('utf-16be', 2);        // 大端序
    }

    // ===== 第二步：尝试 UTF-8 =====
    const utf8Str = buffer.toString('utf-8');
    // 如果包含 U+FFFD（替换字符），说明有无效的 UTF-8 序列
    if (!utf8Str.includes('\uFFFD')) {
      return utf8Str;                                // 纯 UTF-8，没问题
    }

    // ===== 第三步：降级到 GBK =====
    try {
      const iconv = require('iconv-lite');
      return iconv.decode(buffer, 'gbk');             // 用 iconv-lite 转 GBK
    } catch (e) {
      // iconv-lite 未安装时返回最佳努力结果
      logger.warn('[FileController] iconv-lite not available');
      return utf8Str;                                  // 至少不会崩溃
    }
  } catch {
    return null;                                       // 文件不存在或无法读取
  }
}
```

### 检测优先级链路

```
读入文件的原始字节
  ↓
有 BOM 吗？
  ├── EF BB BF → UTF-8 ✅
  ├── FF FE → UTF-16 LE ✅
  └── FE FF → UTF-16 BE ✅
  ↓ 无 BOM
当作 UTF-8 解码
  ↓ 包含 � (U+FFFD)？
  ├── 不包含 → 就是 UTF-8 ✅
  └── 包含 → 尝试 GBK（需 iconv-lite）
       ├── 有 iconv-lite → GBK ✅
       └── 没有 → 返回带乱码的 UTF-8 ⚠️（至少不崩）
```

### 为什么不直接用 chardet 库？

| 因素 | 选择 |
|------|------|
| **准确率** | BOM + FFD 检测对中文场景够用了（>95%）|
| **依赖** | iconv-lite 是可选依赖，没有也不影响运行 |
| **性能** | 纯字符串操作，毫秒级完成 |
| **复杂度** | ~30 行 vs 引入一套统计检测库 |

对于 TTS 工具这种场景——**用户导入的文件格式有限、可以接受极少数边界情况降级**——手写检测足够了。

## Q5：路径安全怎么做的？防止目录穿越攻击

**A：** 任何时候接收用户传入的路径都要校验：

```javascript
// service/tts.js — TTSService.generate()
async generate(params) {
  const { outputPath } = params;

  if (outputPath) {
    const normalized = path.normalize(outputPath);
    if (normalized.includes('..')) {
      throw new Error('Invalid output path: path traversal detected');
    }
  }
  // ...
}
```

```
正常路径:
  /Users/xxx/project/NPC张三/audio → normalize 后不变 ✅

攻击路径:
  ../../../../../etc/passwd → normalize 后包含 '..' ❌ 抛出异常

混合路径:
  /Users/xxx/project/../../etc/hosts → normalize 后包含 '..' ❌
```

这是 Electron 安全模型的**最后一道防线**——即使 preload bridge 暴露的 API 被滥用，路径规范化也能阻止对敏感文件的访问。

## 总结

| 设计决策 | 方案 | 解决的问题 |
|---------|------|-----------|
| 数据存储 | 角色即文件夹（voice.json + script.txt + meta.json + audio/） | 可见、可编辑、可版本控制 |
| 启动恢复 | 扫描目录 → 读取 JSON+TXT → 组装内存对象 | 关闭后再打开不丢进度 |
| 防抖落盘 | 1 秒 debounce + before-quit 兜底 | 避免频繁 IO，保证不丢数据 |
| 编码检测 | BOM 优先 → UTF-8 尝试 → FFD 检查 → GBK 降级 | 中文脚本导入乱码问题 |
| 路径安全 | `path.normalize()` + `..` 检测 | 目录穿越攻击防护 |
